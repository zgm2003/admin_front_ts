# 前端接口集成硬规则：文档即唯一契约

本文件约束本项目后续所有接口接入、字段映射、页面重构和联调工作。Codex、AI Agent 与开发者在修改运行时代码前，必须先阅读本文件。

## 1. 最高原则

后端最新正式文档是前端实现接口契约的唯一来源。

- 文档定义什么接口、参数和字段，前端就只使用这些接口、参数和字段。
- 文档没有定义的字段、别名、优先级、关联关系和空值语义，前端不得自行推断。
- 真实响应与文档不一致时，视为契约问题，不得通过前端兜底把问题隐藏起来。
- 没有最新正式文档，或文档仍在变更时，停止对应接口实现并向用户确认。
- 用户提供的口头说明、截图或单次响应只能用于发现问题，不能代替最终正式文档；除非用户明确要求将该说明作为当前契约。

一句话规则：**文档写什么就用什么；文档没写就不猜、不兼容、不兜底。**

## 2. 契约来源与优先级

按以下顺序判断契约：

1. 用户明确指定的最新后端接口文档。
2. 用户在当前任务中明确确认的补充契约。
3. 仓库内与该接口对应、且能确认仍有效的后端文档。

以下内容不得作为接口契约来源：

- 旧代码中的字段读取逻辑。
- 旧测试、旧 mock、旧静态数据。
- 设计稿、截图或演示数据。
- 某一次偶然返回的响应样例。
- 字段名相似、数值相同或“看起来应该是”的推断。
- 其他项目、其他接口或其他页面的实现。

如果多个文档互相冲突，必须停止实现并要求用户指定唯一有效版本，不得自行选择。

## 3. 严禁猜字段和兜底

接口集成中禁止以下写法和等价行为：

```ts
const id = item.detail_id || item.media_news_id || item.id
const title = item.analysis_title || item.title || '默认标题'
const summary = item.ai_summary || item.summary || item.description
```

禁止行为包括但不限于：

1. 禁止用 `a || b || c` 或 `a ?? b ?? c` 猜测同一业务字段。
2. 禁止因为两个 ID 当前数值相同，就认为它们语义相同或可以混用。
3. 禁止为新接口保留旧字段兼容，除非最新正式文档明确要求兼容。
4. 禁止使用未写入文档的别名字段、备用字段或推测字段。
5. 禁止根据普通字段生成缺失业务数据，例如根据标题生成摘要、根据地区生成政策结论。
6. 禁止把普通 `title`、`summary`、`description` 伪装成 AI 解读或分析正文。
7. 禁止通过默认对象、空数组填充、静态列表或样例数据掩盖响应结构错误。
8. 禁止在接口失败时继续展示上一次数据、mock 数据或硬编码业务内容，除非文档明确规定缓存策略。
9. 禁止把后端错误吞掉并转成“暂无数据”。错误和空数据是两个不同状态。
10. 禁止为了让页面“先能跑”而创建未被文档定义的兼容层。

## 4. 字段必须一对一映射

每个前端字段必须能追溯到文档中的一个明确字段。

允许：

```ts
return {
  id: response.id,
  title: response.title,
  summary: response.summary,
}
```

不允许：

```ts
return {
  id: response.detail_id ?? response.id,
  title: response.analysis_title ?? response.title,
  summary: response.ai_summary ?? response.summary,
}
```

如果页面确实需要不同语义的多个 ID，必须：

- 按文档分别命名。
- 分别保存。
- 分别传给文档指定的接口。
- 用数值不同的测试数据证明没有混用。

不能使用模糊名称掩盖语义，例如在存在多个 ID 时只保留 `id`、`itemId` 或 `targetId`。

## 5. 文档未定义时的处理

遇到以下任一情况，必须停止相关实现：

- 字段不存在于文档。
- 字段含义不清楚。
- 列表 ID 与详情 ID 的关系不明确。
- 同一路径参数在不同接口中的含义不明确。
- 字段是否必填、是否可空不明确。
- 列表接口与详情接口的数据关联方式不明确。
- AI 内容字段与普通内容字段的界限不明确。
- 文档示例、真实响应和口头说明互相冲突。

停止后需要向用户提供：

- 接口方法和路径。
- 不明确的字段名。
- 文档原文或缺失位置。
- 真实响应样例。
- 当前页面需要该字段完成什么行为。

在得到明确契约之前，只允许保留 loading、空态、缺失态或错误态，不允许猜测实现。

## 6. 页面状态规则

运行时只允许以下真实状态：

- `loading`：请求尚未完成。
- `success`：响应符合文档，展示真实数据。
- `empty`：接口按文档成功返回空集合或明确无数据。
- `missing`：响应成功，但缺少文档规定的必要字段。
- `error`：请求失败或响应结构违反文档。

不得：

- 用空态覆盖接口错误。
- 用默认文案覆盖必要字段缺失。
- 用假数据填满空页面。
- 用旧数据假装本次请求成功。
- 用前端生成内容替代后端缺失内容。

后端返回的错误信息在安全且对用户有意义时应直接展示；响应结构不符合文档时，应明确暴露契约错误。

## 7. 共享模块与系统级重构

当首页、行情页或其他页面展示同一业务模块时：

- 共享同一套接口契约、类型、数据适配、详情行为和错误状态。
- 共享逻辑必须位于共同的 API、store、composable 或全局组件中。
- 页面只能通过明确参数表达展示差异，例如条数、列数、是否搜索、是否分页。
- 禁止复制一份模块后分别增加字段兜底。
- 禁止首页与完整页对同一字段做不同解释。
- 禁止一个页面用真实接口，另一个页面继续使用 mock 或硬编码数据。

是否共享组件、共享状态或仅共享数据逻辑，必须依据正式需求和文档确定；不能因为现有页面“看起来一样”就擅自决定状态生命周期。

## 8. API、适配层与组件职责

### API 层

- 只声明文档存在的接口、参数和响应类型。
- 不添加文档未定义的请求参数。
- 不自动改写 ID、kind、分页游标或筛选条件。

### 适配层 / Store / Composable

- 只做文档字段到前端视图字段的一对一转换。
- 可以做纯展示格式化，例如明确日期字段的显示格式。
- 不得合并多个候选业务字段。
- 必要字段缺失时返回明确错误，不得生成默认业务数据。

### 组件层

- 只渲染已经验证的视图模型。
- 不理解后端原始字段优先级。
- 不写字段兼容链。
- 不生成业务结论、AI 内容或缺失数据。

## 9. Mock、测试数据和静态数据

Mock 只允许存在于自动化测试中，并且必须严格模拟文档契约。

禁止：

- 在运行时 UI 中引入 mock。
- 接口失败后回退到 mock。
- 用设计稿数据作为线上默认数据。
- 用硬编码列表验证“页面看起来正常”。
- 测试只使用 ID 恰好相等、字段永不为空的理想样例。

测试必须主动使用能暴露混用的问题数据，例如：

```ts
{
  id: 9301,
  detail_id: 301,
}
```

但只有在正式文档明确存在 `id` 和 `detail_id` 及其关系时，才允许编写上述契约测试。

## 10. 接口实施前检查清单

开始写代码前必须确认：

1. 最新正式文档的文件或链接。
2. 接口方法与完整路径。
3. 路径参数、查询参数和请求体字段。
4. 响应字段、类型、必填性和可空性。
5. 列表项与详情数据的关联字段。
6. 每一个 ID 的独立业务语义。
7. AI 字段与普通内容字段的明确边界。
8. 分页、排序、筛选和搜索契约。
9. 空数据响应。
10. 错误响应及状态码。

任意一项缺失且会影响实现时，不得继续猜测。

## 11. 测试要求

每次接口接入或重构必须先写契约测试，并至少覆盖：

- 正常完整响应。
- 文档允许的空值。
- 必要字段缺失。
- 空集合。
- 接口错误。
- 不同语义 ID 使用不同数值。
- 列表到详情的参数传递。
- 首页与完整页共享模块的一致行为。
- 不存在旧字段兼容和运行时 mock。

测试断言必须验证准确请求路径、准确参数和准确字段，不得只断言“页面出现了内容”。

## 12. 验收要求

接口相关任务完成前必须确认：

1. 实现与最新正式文档逐项一致。
2. 没有未文档化字段。
3. 没有猜测性 `||`、`??` 兼容链。
4. 没有运行时 mock、硬编码业务数据或设计稿兜底。
5. 没有用普通内容冒充 AI 内容。
6. 首页与完整页对同一业务契约解释一致。
7. loading、empty、missing、error 状态可区分。
8. 契约测试覆盖不同语义 ID 和字段缺失。
9. 相关测试通过。
10. `npm run build` 通过。

## 13. Agent 执行要求

Agent 接到接口集成或系统重构任务时必须：

1. 先读取本文件、`AGENTS.md` 和 `docs/agent/` 要求。
2. 找到用户指定的最新正式后端文档并完整阅读相关章节。
3. 列出接口与字段映射后再开始实现。
4. 发现歧义立即停止对应部分并询问用户。
5. 不得以“兼容旧数据”“提高健壮性”“先跑起来”为理由添加兜底。
6. 不得擅自扩大接口契约。
7. 实现后逐项审计文档、类型、请求参数、适配逻辑和测试。

## 最终红线

宁可页面明确报错、显示缺失状态或暂时不实现，也不能让前端猜字段、编数据、做静默兼容或用兜底掩盖后端契约问题。
